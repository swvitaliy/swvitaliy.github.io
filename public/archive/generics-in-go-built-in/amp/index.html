Generics in Go and built-in types

swvitaliy - 23 окт. 2022

Есть такая проблема с генериками в го, что ты не можешь использовать в общих функциях одновременно свои и встроенные типы. Чтобы проиллюстрировать проблему приведу такой пример:

В данном примере описан интерфейс Ordered и функции Min, MinN, Max, MaxN для работы над значениями этого интерфейса. Теперь,  если мы попробуем запихать туда какое-нибудь число или строку, то получим ошибку компиляции (поскольку ни число ни строка не поддерживают этот интерфейс не поддерживают).

Чтобы решить эту проблему, можно сделать отдельный интерфейс для операций, необходимых для работы алгоритма над произвольным типом и для операций над встроенными типами сделать обертку:

Здесь интерфейс Comparator реализуют структуры GreatOrdered, LessOrdered над генеративным типом данных constraints.Ordered, что позволяет использовать общие функции Best, BestN для получения максимума/минимума над числами и строками. Поскольку в go нет SFINAE, то для более конкрентых типов данных нужно делать функции-обертки.

В случае, когда интерфейс операций над значениями содержит один метод, то удобнее описать этот метод в виде функции. Например, как тут:

type Comparator[T any] func(a, b T) bool

func UpperBound[T any](arr []T, v T, less Comparator[T]) int {
	l, r, m := 0, len(arr), 0

	for l < r {
		m = (l + r) / 2
		if less(v, arr[m]) {
			r = m
		} else {
			l = m + 1
		}
	}
	return l
}

Что касается производительности остается надеятся, что компилятор go не будет на каждый вызов Compare для встроенных типов делать реальный вызов, а поймет, что нужно подставить операцию, указанную в return.

Bounds

Чтобы получить MinInt/MaxInt (и крайних значений для типов других) должна быть поддержка передачи в параметры генериков значений. Либо считать константы, зависящие от типов, либо constexpr функции. Пока кажется, что таких возможностей в языке нет (go 1.19). Ведется обсуждение в открытом пропосале https://github.com/golang/go/issues/50019. Поэтому, применить тот же принцип, что и выше и построить конструкцию вроде такой не получится:

type Bounds[T any] interface {
	MinValue() T
	MaxValue() T
}

type OrderedBounds[T constraints.Ordered] struct {
}

func (b OrderedBounds[T]) MaxValue() T {
	// return ???
}

func (b OrderedBounds[T]) MinValue() T {
	// return ???
}

Если же делать для каждого типа – Int32Bounds, UInt64Bounds и т.д., то придется передавать их в вызов каждый раз:

type Bounds[T any] interface {
	MinValue() T
	MaxValue() T
}

type IntBounds struct {
}

func (b IntBounds) MaxValue() int {
	return math.MaxInt
}

func (b IntBounds) MinValue() int {
	return math.MinInt
}

// ...

func BestN[T any, C Comparator[T]](list []T, c C, init T) T {
	return Reduce(list, func(ret, v T) T {
		return Tern(c.Compare(ret, v), ret, v)
	}, init)
}

func MaxNOrdered[T constraints.Ordered](list []T, b Bounds[T]) T {
	return BestN(list, GreatOrdered[T]{}, b.MinValue())
}

func MinNOrdered[T constraints.Ordered](list []T, b Bounds[T]) T {
	return BestN(list, LessOrdered[T]{}, b.MaxValue())
}

func main() {
	a := []int{1, 2, 23, -5, 3}
	minVal := MinNOrdered(a, IntBounds{})

	// ...
}

Это не очень здорово выглядит и в более сложных случаях кажется, что нужно будет делать свои классы с метаинформацией о типе и передавать их на вызов генерик функции. Надеюсь, что разработчики языка примут какое-то решение о получении данных о типе на этапе компиляции, что позволит сильно упростить код и не переводить накладные расходы из compile-time в runtime, как в примере выше.
