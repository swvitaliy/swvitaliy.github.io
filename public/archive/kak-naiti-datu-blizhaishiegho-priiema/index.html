Как найти дату ближайшего приема?

3 янв. 2021 г.

В данной заметке описано принципиальное решение поиска ДБП (Даты Ближайшего Приема) в системе хранения расписания ресурсов в Redis.

Проблема

В заданной системе расписание ресурсов (работников, оборудования и др.) хранится в редис кластере. В качестве ключа там выступают идентификатор бизнеса и идентификатор ресурса. В качестве значения выступает вектор, в котором содержатся 0 и 1 (занято время или свободно) с шагом в 1 или 5 минут. Так же, если расписание зависит от таксономии (услуги), то для каждой из таксономий содержится еще и свой вектор расписания с ключом содержащим идентификатор бизнеса, идентификатор ресурса и идентицикатор таксономии (вектор без идентификатора таксономии в таком случае содержит объединяющий вектор по всем таксономиям).

Для того, чтобы получить ДБП (Дату Ближайшего Приема) для работника напрямую нужно произвести поиск, начиная с текущей даты. Суть этого поиска в том, что у работника за день могут быть рабочие интервалы меньшие, чем длительность услуги. Поэтому, если сравнить длительность услуги с длительностью самого длинного рабочего интервала за день, то можно сказать может ли этот работник выполнить услугу в этот день или нет. Ближайший к текущей дате такой день и есть ДБП.

Кроме того, данный индекс должен отвечать на следующие запросы:

  1. найти ДБП по работнику в бизнесе
  2. найти ДБП по работнику для заданной услуги
  3. найти работников, которые могут выполнять заданную услугу (услуги)

Для первой и второй задачи линейный поиск худо-бедно справляется, однако для 3-й задачи необходимо выполнить линейный уже по набору работников, что не представляется возможным по соображениям производительности. Тем более, что этот запрос может быть не только в рамках бизнеса, но и в рамках филиальной сети и даже произвольного достаточно большого набора бизнесов (сквозной поиск).

Запрос должен быть выполнен за короткое время (~100 мс), поэтому тут нужна максимум логарифмическая вычислительная сложность на небольшом объеме данных.

Принципиальное решение

Можем воспользоваться тем фактом, что данные, получаемые из SORTED SET с помощью команды ZRANGEBYSCOREвозвращаются в лексикографическом порядке и сохранять в качестве значений строки "ДБП|res_id". В качестве весов брать длительности свободных интервалов (всех свободных интервалов за каждый день).

zfree:business_id1:tax_id1
    20:2020-08-01T18:15:00Z|res_id1
    20:2020-08-01T18:20:00Z|res_id2
    30:2020-08-01T16:00:00Z|res_id5
    30:2020-08-01T20:10:00Z|res_id1
    ...

Для одного работника в этой структуре может быть несколько значений. Например, для res_id1 интервал длительностью 20 минут 2020-08-01T18:15:00Z, длительностью 30 минут 2020-08-01T20:10:00Z.

Тогда список ресурсов, упорядоченных по ДБП, с длительностью >= tax_duration:

ZRANGEBYSCORE zfree:business_id1:tax_id1 tax_duration +inf WITHSCORES LIMIT 0, 10

В полученной выборке может потребоваться отфильтровать первые несколько элементов, тк. дата этих интервалов уже устарела. С помощью команды ZREM можно почистить эти значения.

Список ДБП без учета таксономии сохраним отдельно по ключу zfree:business_id1.

Если нужно будет получить ДБП по одному конкретному работнику можно использовать так же SORTED SET в редис.
